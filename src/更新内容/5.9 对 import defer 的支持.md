TypeScript 5.9 引入了对ECMAScript 延迟模块评估提案的支持，采用了新的`import defer`
语法。该功能允许你导入模块而不立即执行该模块及其依赖项，从而能更好地控制任务执行和副作用发生的时机。

该语法仅支持命名空间导入：

```typescript
import * as feature from "./some-feature.js";
```

`import defer`的核心优势在于，模块仅在首次访问其某个导出项时才会被求值。请看这个示例：

```typescript
// ./some-feature.ts
initializationWithSideEffects();

function initializationWithSideEffects() {
  // ...
  specialConstant = 42;

  console.log("Side effects have occurred!");
}

export let specialConstant: number;
```

使用 `import defer` 时，`initializationWithSideEffects()` 函数要到你实际访问导入命名空间的某个属性时才会被调用：

```typescript
import * as feature from "./some-feature.js";

// No side effects have occurred yet

// ...

// As soon as `specialConstant` is accessed, the contents of the `feature`
// module are run and side effects have taken place.
console.log(feature.specialConstant); // 42
```

由于模块的评估会被推迟到你访问该模块的成员之后，因此你无法将命名导入或默认导入与`import defer`一起使用：

```typescript
// ❌ Not allowed
import { doSomething } from "some-module";

// ❌ Not allowed
import defaultExport from "some-module";

// ✅ Only this syntax is supported
import * as feature from "some-module";
```

请注意，当你编写 `import defer` 时，模块及其依赖项会被完全加载并准备好执行。这意味着该模块必须存在，并将从文件系统或网络资源中加载。常规
`import` 与 `import defer` 的关键区别在于，语句和声明的执行会被延迟，直到你访问导入的命名空间的属性。

此功能对于有条件加载初始化成本高昂或特定于平台的模块特别有用。它还可以通过将应用功能的模块评估推迟到实际需要时再进行，来提升启动性能。

请注意，TypeScript 完全不会对`import defer`进行转换或“降级”处理。它旨在用于原生支持该特性的运行时，或由诸如打包工具等能够执行相应转换的工具来处理。
这意味着`import defer`仅在`--module`的`preserve`和`esnext`模式下才能生效。
